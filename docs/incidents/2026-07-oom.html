ROOOT · Stands service · Incident report

The stands service ran out of memory for three nights. Here is why, and the fix that shipped.

It was never the crowd, the feed, or bad luck. The service simply never let go of a match once it ended — so every game we played stacked on top of the last until the machine was killed. That is now fixed at the root and live in production.

What you saw

Three nights of the same failure: mid-match, the service stopped accepting new connections and health checks, then the machine restarted itself. Each crash carried the Linux out-of-memory signature — exit code 137, oom_killed=true. Last night it compounded: while the service thrashed, a footage-capture tool I built wrote an event log with no size limit and grew it to roughly 200 GB, filling a quarter of the disk. Deleting that log freed the disk; it was a symptom of the same lesson, not the disease.

Root cause

The service never evicted a finished match.

A match created at kickoff stayed fully resident in memory forever — its market-tick buffers, its join-replay history, its sentiment accumulator, and eleven per-match caches. Nothing ever removed it. Three games across three days on a process that only restarted when it crashed meant three games' worth of state piled up on a 256 MB guest until the kernel killed it. The earlier fixes — swap, a self-probe, a liveness watchdog — softened the fall but never stopped the climb.

The soak test that had reported a "flat heap" replayed a single match quickly. It never modelled several games persisting over many hours, which was the only thing that mattered. That gap is why this recurred.

The fix

  1. Evict finished matches

    A resolved match with no one watching and no activity for 15 minutes (or any match idle for 3 hours) is now swept out of the registry and all eleven per-match caches through a single cleanup seam. What is persisted — the durable sentiment record on the volume — is untouched; only the in-memory bulk is released.

  2. Make settlement idempotent on disk

    A match is never crystallized or anchored twice: before writing, the service checks the volume for an existing record and skips both the write and the on-chain anchor if one is there. This survives eviction, restart, and lost in-memory state.

  3. Count every watcher in the evict gate

    The "no one watching" check now counts feed-only spectators too, so a finished match someone is still viewing can never be evicted out from under them.

The bug the fix almost introduced — caught in review

Eviction, on its own, re-opened an old hole: an evicted match dropped from the snapshot could be re-settled and anchored on-chain a second time after a restart — the same defect that produced a duplicate record earlier this week. Review reproduced it before deploy (two records, two hashes). Step 2 above closes it permanently, and a new test locks it: it fails on the old code, passes on the new.
