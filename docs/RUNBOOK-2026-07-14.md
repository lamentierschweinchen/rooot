## 3 · Failure playbook (symptom → diagnose → act → abort)

During the freeze you observe and escalate. Deploys/restarts are the coordinator's and are
live-severity only — a restart is itself a cost (§3b). Screenshot + timestamp everything.

**(a) SSE feed drops mid-match**
- *Symptom:* odds/score stop updating; a lens looks frozen.
- *Diagnose:* re-run the §2.3 feed probe; `curl` health (§2.1) for uptime. Add
  `if(m.type==='feedState')console.log(m)` to the probe to see the feed state.
- *Reality:* reconnect is automatic — on the client adapters AND the service's upstream SSE.
  After any blip TxLINE SSE takes up to ~2 min to reattach.
- *Act:* **WAIT.** Watch the probe for ticks resuming. Nothing to touch.
- *DO NOT* restart/redeploy to "kick" the feed — that resets the crowd's roar to silence,
  drops presence, and risks the lineup catch, to fix something that self-heals.
- *Abort/escalate:* silence past ~3 min with the service up (uptime climbing, health OK) →
  escalate coordinator with the probe output + timestamp. Health unreachable > 5 min → §3b.

**(b) OOM / restart mid-match**
- *Symptom:* health briefly unreachable, then `uptime` resets to seconds; `matchesActive` /
  `clients` drop.
- *This is now survivable by design:* eviction stopped the leak; crystallize is
  disk-idempotent (no double-anchor on the re-settle path); the roster re-seeds from the
  scores snapshot on boot; verdicts replay on reload. Roar deliberately resets to silence
  (stale decay is not resurrected).
- *What a restart costs:* live roar/presence reset (the cheer bar returns to silence and
  rebuilds from live taps); ~2 min SSE warm-up before odds resume; any fan mid-Collect
  retries.
- *Verify recovery, in order:*
  1. `curl -s https://rooot-stands.fly.dev/health` → back up, `uptime` climbing.
  2. Odds resume on the §2.3 probe within ~2 min (warm-up rule).
  3. **THE ONE THING: team sheet + clock.** Open `https://rooot.club/stadium.html` — the
     TEAM SHEET still shows real names (boot seed recovered the roster) and the clock/score is
     right. Confirm the seed fired:
     ```
     flyctl logs -a rooot-stands | grep "txline:seed] lineup seed"
     ```
     GREEN = `[txline:seed] lineup seed 18237038: roster recovered (N players …)`.
- *Act:* all three green → log it and move on, no action. Team sheet EMPTY after restart AND
  no seed line (or it says the envelope failed to parse) → escalate coordinator (the snapshot
  lacked the one-shot; honest empty is correct — never fabricate names).
- *Abort/escalate:* machine restart-loops (uptime never climbs past ~120s across 3 checks) →
  escalate coordinator/owner, service down.

**(c) Vercel frontend bad** (wrong render, broken page, bad asset)
- *Symptom:* rooot.club serves a broken/wrong-fixture page; smoke (§2.4) rows fail on a live
  route.
- *Diagnose:* `vercel ls` (list recent deployments; note the last-known-good URL). Re-run §2.4.
- *Act (coordinator):* roll back to the prior good deployment —
  ```
  vercel rollback <previous-deployment-url> --yes
  ```
  (`vercel rollback` with no target reverts to the immediately previous deploy;
  `vercel rollback status` shows progress.) Re-run §2.4 — must be green.
- *Abort:* not restored within ~10 min → escalate owner. The frontend is static — a bad
  frontend never corrupts the record or the crowd; the service keeps running.

**(d) Fly bad deploy** (service broken right after a service deploy)
- *Symptom:* health failing or the service misbehaving immediately after a deploy.
- *Diagnose:* `flyctl releases -a rooot-stands` (add `--image` for image refs) — find the last
  good release + its image.
- *Act (coordinator):* this flyctl (v0.4.69) has **no one-shot rollback** — redeploy the prior
  image:
  ```
  flyctl releases --image -a rooot-stands              # copy the last-good image ref
  flyctl deploy --config services/stands/fly.toml --image <that-ref>
  ```
  A redeploy **RESTARTS the machine** — treat it as §3b (verify team sheet + clock after). Only
  do this if the current state is worse than a restart.
- *Abort:* if the prior image also fails health, escalate owner — do not thrash deploys during
  a live match.

**(e) Mint fails during Collect**
- *Symptom:* a fan taps Collect (`terrace.html`) and the scarf doesn't mint.
- *What the fan sees (by design):* an honest **"Try again"** — nothing lost; the seat and
  record are intact; Collect is idempotent and full-time-gated, so a retry re-attempts the
  same mint, never a double.
- *Diagnose:* `flyctl logs -a rooot-stands | grep -Ei "\[mint\]|irys|collect"`. Two usual
  causes: (1) mint payer low/empty → check balance (§2.7), refill (§2.7); (2) RPC/DAS — the
  album read needs `HELIUS_RPC_URL` (a Fly secret); the public devnet RPC has no DAS index, so
  a mint that lands but doesn't appear in the album is DAS lag / HELIUS, not a lost mint.
- *Act:* refill the payer if low (owner/coordinator); the fan retries Collect. No restart
  needed for a refill — the balance is read per-claim.
- *Abort:* persistent failure with the payer funded and RPC healthy → escalate coordinator.
  The fan's "Try again" holds honestly, so this is not a fire — capture the log line for
  post-match.

**(f) Wrong / empty fixture on the site** — PRE-FREEZE ONLY
- *Symptom:* rooot.club or `/live` shows the wrong teams; §2.5 fixture identity is RED.
- *Diagnose:* `cat apps/web/public/fixture.json` (`matchId` must be `18237038`, FRA/ESP);
  `grep -c "match=18237038" vercel.json`.
- *Act (coordinator, before 18:00 only):* re-stage + redeploy —
  ```
  cd /Users/ls/Documents/rooot && node scripts/cutover-fixture.mjs 18237038
  git diff apps/web/public/fixture.json vercel.json    # only matchId/home/away/colors/kickoff + /live pin change
  git add apps/web/public/fixture.json vercel.json && git commit -m "cutover: FRA-ESP 18237038" && git push origin main
  # deploy — coordinator's method; documented pattern (VERIFY-BEFORE-USE with coordinator):
  git worktree add /tmp/rooot-deploy origin/main && cp -r .vercel /tmp/rooot-deploy/ && cd /tmp/rooot-deploy && vercel --prod --yes
  ```
  Re-run §2.4 + §2.5.
- *Abort:* after 18:00 the fixture is frozen. A wrong fixture discovered mid-match IS
  live-severity — escalate owner + coordinator immediately; a cutover mid-match restarts
  nothing on Fly but does redeploy the frontend and must be a conscious owner call.
